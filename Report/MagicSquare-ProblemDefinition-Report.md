## STEP 1 — Observation (관찰)

### 1.1 현재 우리가 해결하려는 상황은 무엇인가?

**관찰:**  
누군가는 **1부터 16까지의 정수를 격자 한 칸에 하나씩 배치**해야 하는 상황에 있다. 배치가 끝나면, **행·열·대각선마다 합이 동일한지**를 확인하거나, 그 조건을 만족하는 배치를 찾아야 한다.

- **입력/출력:** 4×4 격자와 16개의 서로 다른 숫자
- **규칙:** 여러 방향(가로, 세로, 대각)에서 **합이 하나의 값으로 맞아야 함**
- **불확실성:** 조건을 만족하는 배치가 **하나인지, 여러 개인지**, 사용자가 **직접 채울지·자동으로 찾을지**는 아직 명시되지 않음
- **맥락:** `DEV\MagicSquare` 경로는 **학습·실습용 개발 환경** 안에서의 과제 또는 연습 과제로 보임

**한 줄 정리:**  
16개 숫자를 4×4에 놓았을 때, 여러 줄의 합이 일치하는지(또는 그렇게 놓는 방법이 있는지)를 다루는 문제에 가깝다.

### 1.2 왜 4×4 마방진 문제를 다루는가?

| 관점 | 관찰 내용 |
|------|-----------|
| **규모** | 3×3보다 제약이 많고, 5×5 이상보다 **수작업·머리로 검증하기 어렵지만**, 프로그램으로는 여전히 **전수·규칙 탐색이 가능한 크기** |
| **수학** | 홀수/짝수 차수, **짝수 차수(4×4)** 전용 구성법이 따로 있어, “한 가지 공식”이 아닌 **케이스 분기**를 경험하기 좋음 |
| **검증** | 마법 상수(4×4에서는 **34**)가 고정되어, **합 하나로 맞는지 틀리는지** 판별이 단순함 |
| **프로그래밍 학습** | 2차원 배열, 반복문, 조건 분기, (선택) 재귀/백트래킹, 입출력 포맷 등 **기초 구조를 한 과제에 묶기** 적합 |
| **이름/폴더** | “Magic Square”는 **고전 퍼즐 + CS 입문 과제**로 널리 쓰이는 레이블 |

### 1.3 이 문제는 어떤 학습 또는 시스템 설계 맥락에서 등장하는가?

**학습 맥락**

- 알고리즘 입문: 배열 순회, 합 계산, 불변 조건 검사
- 문제 분해: “유효한가?” vs “하나를 만들어라” vs “모두 나열해라”는 **서로 다른 과제**
- 추상화 연습: 격자 = 2D 구조, “마법 상수” = 도메인 상수, “유효성” = 순수 함수로 분리하기 쉬움
- 디버깅: 잘못된 한 칸만 바꿔도 **여러 줄의 합이 동시에 깨짐** → 테스트·시각화의 필요성이 자연스럽게 드러남

**시스템·설계 맥락 (등장할 수 있는 맥락)**

- 단일 책임: 검증기 / 생성기 / UI(격자 출력)를 나누는 **모듈 경계** 연습
- 도메인 규칙: “합이 같다”는 **비즈니스 규칙**을 코드 한곳에 모으는 패턴
- 확장성 질문: 나중에 3×3, 5×5로 바꾸면 **차수(n)를 파라미터**로 둘지
- **사용자 시나리오 미정:** 교사용 채점기, 학생용 연습 도구, CLI 데모, 게임 퍼즐 등

### 1.4 상황을 관찰 관점으로 서술 (대체 표현)

> **16개의 서로 다른 정수를 4×4 격자에 배치했을 때, 정해진 여러 선(행·열·대각) 위 숫자의 합이 모두 같아지는지를 사람이 확인하거나, 그 조건을 만족하는 배치를 찾는 활동**을, 학습·실습 환경에서 프로그램으로 다루려는 상황이다.

### 1.5 관찰 단계에서 아직 보이지 않는 것

- 최종 목표가 **검증만**인지, **생성·해 찾기**인지, **사용자 입력 퍼즐**인지
- **언어·실행 환경**(콘솔, 웹, GUI)과 **사용자**
- **하나의 정답**을 보여주면 되는지, **모든 해** 또는 **랜덤 한 해**가 필요한지

## STEP 2 — Why #1 (왜 완성해야 하는가)

### 2.1 질문

**Q: 왜 마방진을 완성해야 하는가?**

### 2.2 가정 A

> **“완전한 4×4 배치”는 과제·시험·자동 채점에서 ‘통과’의 유일한 기준이며, 미완성 격자는 제출·실행 결과로 인정되지 않기 때문에 완성이 필요하다.**

즉, 완성은 사용자의 호기심이 아니라 **외부 판정 시스템(루브릭, 테스트, 채점 스크립트)이 요구하는 종료 상태**다.

### 2.3 가정 A에서 드러나는 불편함

| 불편 | 설명 |
|------|------|
| **“완성”의 의미가 한 가지가 아님** | 칸 채움 / 합 일치 / 숫자 집합(1~16) / 출력 형식이 겹침 → “다 채웠다”≠“맞다” |
| **완성 압력 → 수단·목표 뒤바뀜** | 알고리즘 이해보다 **알려진 정답 배치 출력**이 유리해질 수 있음 |
| **진행 상태가 보이지 않음** | 14칸 맞고 2칸 틀림 → 시스템에는 0과 100만 존재하는 이분법 |
| **“하나 완성” vs 해의 다양성** | 유효 배치가 여러 개일 수 있음 → 채점 기준(특정 배치 vs 아무 유효 배치) 불명확 |
| **완성 주체 모호** | 사람 / 프로그램 / 협업 — 같은 “완성”이라도 **다른 시스템**을 의미 |

### 2.4 가정 A에서 드러나는 구조적 문제

| 구조 | 내용 |
|------|------|
| **목표가 “상태”이지 “능력”이 아님** | 채점은 최종 격자만 보며, 검증·생성·설명 능력이 요구사항에 안 드러남 |
| **검증과 생성의 분리 부재** | validator 없이 generator만 있거나, 그 반대 |
| **도메인 규칙의 분산** | PDF·구두·테스트·구현자 머릿속에 기준이 조금씩 다름 |
| **완성=종료 vs 학습=과정** | 과정 데이터(시도, 틀린 합)가 인정되지 않으면 교육 목표와 채점 목표가 위장됨 |

### 2.5 Why #1 요약

| 항목 | 내용 |
|------|------|
| **가정 A** | 외부 판정이 **유효한 전체 배치**만 성공으로 인정 |
| **핵심 불편** | 완성은 이분법인데 작업은 다단계; 채움≠정답; 해 다수; 주체 불명확 |
| **핵심 구조 문제** | **완성(상태)**이 **검증·생성·설명(능력)**을 가리고, **완성 기준의 단일 정의** 부재 |

## STEP 3 — Why #2 (왜 프로그램인가)

### 3.1 질문

**Q: 왜 단순 계산이 아니라 프로그램으로 구현하는가?**

### 3.2 반복 가능성 (Repeatability)

**답:** 손풀이는 **그 순간의 결과**일 뿐이다. 프로그램은 **같은 규칙을 N번, 다른 입력·시도에 대해 동일 절차로** 재실행할 수 있다.

**드러나는 문제:** 하드코딩된 정답도 “반복 출력”은 가능 → **규칙 재현**과 **값 복사** 구분 필요. 반복 대상(검증만? 생성? n 변경?)이 요구사항에 없으면 정당화 약함.

### 3.3 검증 자동화 (Automated verification)

**답:** 행·열·대각·숫자 집합 등 **항목이 많아** 사람은 누락·피로 오류가 난다. 프로그램은 **전 제약을 같은 기준으로** 짧은 시간에 검사하고, 통과/실패를 일관되게 보고할 수 있다.

**드러나는 문제:** 검증기 버그는 오류를 **더 빠르게 확정**함. 기준 불일치 시 자동화가 갈등 증폭. `false`만 있으면 **진단 정보** 부재.

### 3.4 오류 방지 (Error prevention)

**답:** 덧셈 실수, 중복 숫자, 한 줄만 확인 등을 **매 실행마다 전체 제약**으로 적용. 부분 완성 상태의 **중간 검사**도 루틴화하기 쉽다.

**드러나는 문제:** 잘못된 규칙도 **일관되게** 틀린 결과를 냄. 검증 없는 생성은 **오류 은폐**. 입력 단계 오류는 검증 전에 이미 깨짐.

### 3.5 규칙 기반 사고 훈련 (Rule-based thinking)

**답:** 자연어 한 줄을 **겹치지 않는 검사 가능한 조건**(값 집합, 행 합, 열 합, 대각, S=34)으로 쪼개고, **선언적 조건**과 **절차적 전략**을 분리하는 연습.

**드러나는 문제:** 암기 절차만 복사, 합만 보고 중복 검사 누락, 과도한 n 일반화. 채점이 출력만 보면 **교육 목표와 채점 신호 불일치**.

### 3.6 Why #2 요약

| 관점 | 프로그램이 주는 것 |
|------|-------------------|
| 반복 가능성 | 동일 절차의 재실행 |
| 검증 자동화 | 전 제약 동시 검사 |
| 오류 방지 | 규칙 고정·중간 검사 (생성+검증+입력 검사 함께일 때) |
| 규칙 기반 사고 | 조건의 명시·분해·조작 |

**한 줄:** 프로그램은 오류를 없애는 것이 아니라, 오류 유형을 **규칙·테스트·구조**로 옮긴다. Why #2 성립에는 **검증·진단·요구 기준의 단일 정의**가 전제된다.

## STEP 4 — Why #3 (TDD 핵심 발견)

### 4.1 질문

**Q: 왜 이 문제를 TDD 방식으로 설계하려 하는가?**

**전제:** TDD는 **맞음의 정의를 실행 가능한 명세로 먼저 고정**하고, 구현은 그 명세를 만족시키는 방식이다.

### 4.2 무엇이 통제되어야 하는가?

| 통제 대상 | 이유 |
|-----------|------|
| “유효한 배치”의 정의 | 행·열·대각·숫자 집합 포함 범위 |
| 마법 상수 S (=34) | 합 비교의 단일 기준 |
| 숫자 집합 1~16 | 합만 맞는 가짜 배치 방지 |
| 격자 차수 n | 범위 확장 시 드리프트 감지 |
| 입력·출력 형태 | 채점·표시와 도메인 분리 |
| 동치 처리 | 회전·대칭·다른 해 허용 여부 |
| 생성 vs 검증 경계 | 한쪽만 통과하는 착각 방지 |

**핵심 발견:** 통제 대상은 **최종 격자 한 장**이 아니라 **“무엇을 유효하다고 부를지”에 대한 계약**이다.

### 4.3 어떤 불변 조건이 존재하는가?

**도메인 불변 (4×4, 표준 정의)**

| ID | 불변 |
|----|------|
| I1 | 격자 4×4 |
| I2 | 16칸 모두 정수로 채움 |
| I3 | 값 집합 {1,…,16} 각 1회 |
| I4 | 각 행 합 = S |
| I5 | 각 열 합 = S |
| I6 | (정의 시) 대각 합 = S |
| I7 | S = 34 |

**모듈·책임 불변**

| ID | 불변 |
|----|------|
| P1 | validate 통과 ⟹ I3∧I4∧I5∧(I6) |
| P2 | validate 실패 ⟹ 제약 중 하나 이상 위반 |
| P3 | 생성 결과는 항상 validate 통과 |
| P4 | 검증은 입력을 변경하지 않음 |
| P5 | 동일 입력 → 동일 검증 결과 |

### 4.4 왜 입력/출력이 명확해야 하는가?

- 테스트는 **함수 계약** — 입출력 고정 없이 테스트 불가
- 출력 애매 시 버그 위치(검증·생성·포맷) 분리 불가
- 도메인 결과와 사람용 표시 혼합 시 **도메인 테스트 오염**
- 입력 스키마(완전 격자 vs 부분 격자)에 따라 **다른 프로그램**

**핵심 발견:** I/O 명확성 = **테스트가 호출할 경계 고정**. 불명확하면 “일단 돌아가게”·하드코딩·과신으로 이어짐.

### 4.5 Why #3 한 줄 요약

4×4 마방진은 규칙이 많고 “완성”이 모호하기 쉬우므로, TDD는 **불변을 테스트로 고정**하고, **I/O 계약으로 모듈을 나눈 뒤**, 구현이 그 계약만 만족하도록 **통제**하려는 설계 선택이다.

### 4.6 다음 단계로 확정할 항목 (미결)

1. 대각(I6) 포함 여부 — 예/아니오  
2. 핵심 API 책임 이름 수준의 분리 (validate / generate / magicConstant)  
3. 입력 스키마: 완전 격자만 vs 부분 격자  
4. 출력: boolean vs 진단 포함 결과  
5. 동치: 특정 격자 고정 vs validate만

## STEP 5 — 진짜 문제 정의

### 5.1 표면 문제 정의 (잘못된 정의)

> **4×4 마방진 프로그램을 만든다. 1부터 16까지 숫자를 넣어 모든 줄의 합이 같게 출력한다.**

| 결함 | 관찰 |
|------|------|
| 목표가 산출물 한 장 | 맞음의 기준·재검사·오류 처리 없음 |
| “모든 줄” 모호 | 대각·부분 줄 불명확 |
| 숫자 규칙 누락 | 합만 맞고 중복·범위 밖 허용 오인 |
| 주체·맥락 없음 | 누가·왜 불명확 |
| 능력 혼동 | 생성·검증·표시·입력 한 덩어리 |
| 이분법 완료 | 어디가 틀렸는지·중간 상태 없음 |

### 5.2 개선된 문제 정의 (정확한 정의)

> **4×4 격자에 대한 “유효 배치”를 도메인 규칙으로 명시하고, 임의의 완전한 배치가 그 규칙을 만족하는지 기계적으로 판정할 수 있어야 한다. 필요 시, 규칙을 만족하는 완전한 배치를 일관된 절차로 도출할 수 있어야 한다. 판정 기준은 과제·채점·실행마다 달라지지 않도록 단일하게 유지한다.**

| 요소 | 내용 |
|------|------|
| 대상 | 4×4, 16칸 모두 채운 정수 배치 |
| 값 집합 | 1~16 각 1회 |
| 선형 제약 | 행·열·(확정 시) 대각 합 = S |
| S | 34 |
| 책임 ① | 완전 배치의 규칙 만족 판정 (필요 시 위반 구분) |
| 책임 ② | (범위 시) 유효 배치의 재현 가능한 도출 |
| 명시적 선택 | 대각 포함, 동치(회전·대칭), 정답 고정 여부 — 시작 전 문서화 |

**한 문장:** “마방진 그림 만들기”가 아니라, **“4×4 유효 배치에 대한 공통 규칙과 판정·(선택) 도출”**이다.

### 5.3 핵심 Invariant

**구조:** G1 (4×4), G2 (16칸 채움)  
**값·집합:** V1 (1~16), V2 (서로 다름), V3 (집합 = {1,…,16})  
**합:** S1 (S=34), S2 (행), S3 (열), S4 (대각, 범위 확정 시)  
**시스템:** C1 (판정 결정·무변경), C2 (도출은 G·V·S 만족), C3 (부분 만족 ≠ 유효)

**우선순위:** G → V → S → C

### 5.4 실제로 훈련하려는 사고 능력

1. **규칙 명시화** — 조건 분해, 과제·채점·구현 기준 일치, 대각 등 사전 확정  
2. **계약·경계 사고** — 판정 vs 도출, 도메인 vs 표시, 정상 vs 잘못된 입력  
3. **불변 중심 검증** — 반례로 각 불변 분리, “거의 맞음” 설명  
4. **재현·통제 (TDD)** — 실행 가능 명세, 우연히 맞는 출력 vs 규칙 통과  
5. **문제 재정의** — 표면 요구 vs 진짜 요구 구분  

**비목표:** 배치 암기, 합만 맞으면 됨, “돌아가 보이면 됨”
